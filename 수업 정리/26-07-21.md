# 2. ROS 2 공통 패키지

ROS 2는 하나의 프로그램으로 구성되는 것이 아니라, 역할이 다른 여러 공통 패키지가 계층적으로 연결되어 동작한다.

## 2.1 주요 공통 패키지

| 구분 | 대표 패키지 | 역할 |
|---|---|---|
| Build system | `ament_cmake`, `ament_package`, `ament_lint` | ROS 2 패키지를 구성하고 빌드하기 위한 기능 |
| RMW | `rmw`, `rmw_implementation` | ROS 2와 DDS 구현체 사이를 연결하는 미들웨어 인터페이스 |
| RCL | `rcl`, `rclcpp`, `rclpy` | ROS 2 기능을 프로그래밍 언어에서 사용할 수 있게 제공 |
| Utility | `rcutils`, `pluginlib`, `class_loader`, `tracetools` | 로그, 플러그인, 클래스 로딩, 추적 등의 보조 기능 |
| Interface pipeline | `rosidl_cmake`, `rosidl_generator_cpp`, `rosidl_generator_py` | 메시지·서비스·액션 인터페이스 코드를 생성 |
| Interface definitions | `std_msgs`, `std_srvs`, `geometry_msgs`, `sensor_msgs` | 자주 사용하는 메시지와 서비스 자료형 제공 |
| Launch | `launch`, `launch_ros`, `launch_xml`, `launch_yaml` | 여러 노드와 설정을 한 번에 실행 |
| Features | `navigation2`, `moveit2`, `tf2`, `sros2` | 내비게이션, 로봇팔 제어, 좌표계, 보안 등의 기능 |
| Robot | `urdf`, `urdfdom` | 로봇의 구조와 형상을 표현 |
| Tools | `rviz2`, `ros2cli`, `rqt`, `webots` | 시각화, 명령어, GUI, 시뮬레이션 도구 |
| Third-party packages | `gtest_vendor`, `tinyxml2_vendor` 등 | ROS 2에서 사용하는 외부 라이브러리 |
| Examples | `demo`, `examples` | 학습과 테스트를 위한 예제 |

> **기억할 점:** ROS 2는 빌드 시스템, 언어 라이브러리, 통신 미들웨어, 메시지 인터페이스 및 개발 도구가 각각 패키지 형태로 나뉘어 있다.

# 3. ROS 2의 중요 개념 및 특징

## 3.1 DDS란?

**DDS(Data Distribution Service)**는 분산 시스템에서 데이터를 주고받기 위한 통신 미들웨어 표준이다.

OMG(Object Management Group)에서 표준화하였으며, 발행자와 구독자가 데이터를 주고받는 **Publisher-Subscriber 방식**을 사용한다.

ROS 2에서는 DDS를 이용하여 노드끼리 통신한다.

```text
사용자 코드
    ↓
rclpy 또는 rclcpp
    ↓
RCL
    ↓
RMW
    ↓
DDS 구현체
    ↓
네트워크 통신
```

### DDS의 특징

- 발행자와 구독자 방식으로 데이터를 주고받는다.
- 노드를 자동으로 발견하는 **동적 검색(Discovery)**을 지원한다.
- 중앙의 `roscore` 없이 노드끼리 통신할 수 있다.
- QoS 설정을 이용하여 통신 신뢰도와 데이터 보관 방식을 조절할 수 있다.
- 작은 임베디드 장치부터 대규모 분산 시스템까지 확장할 수 있다.
- 여러 DDS 구현체 중 하나를 선택하여 사용할 수 있다.

### DDS와 RTPS

DDS의 발행자와 구독자 사이에서 데이터를 전달하기 위해 RTPS 계열의 통신 규약이 사용된다.

```text
RTPS = Real-Time Publish-Subscribe
```

---

## 3.2 RMW란?

**RMW(ROS Middleware Interface)**는 ROS 2와 DDS 구현체 사이를 연결하는 계층이다.

사용자가 작성한 코드가 특정 DDS 회사나 제품의 코드에 직접 의존하지 않도록 중간에서 동일한 인터페이스를 제공한다.

```text
사용자 코드 → ROS 2 API → RMW → DDS 구현체
```

따라서 ROS 2 사용자는 일반적으로 DDS 코드를 직접 작성하지 않고 `rclpy` 또는 `rclcpp`를 사용한다.

---

## 3.3 실시간 제어와 결정적 실행


```text
Real-time control
Deterministic execution
Real-time communication
```

### Real-time control

정해진 시간 안에 센서 데이터를 처리하고 모터 명령을 출력하는 제어 방식이다.

### Deterministic execution

같은 조건에서 작업이 예측 가능한 순서와 시간 안에 실행되는 것을 의미한다.

### Real-time communication

중요한 데이터가 정해진 시간 안에 전달될 수 있도록 통신 지연과 신뢰도를 관리하는 것을 의미한다.

> ROS 2를 사용한다고 모든 프로그램이 자동으로 실시간성을 보장받는 것은 아니다. 실시간 운영체제, Executor 구성, 메모리 관리, 콜백 구조 및 QoS 설정 등을 함께 고려해야 한다.

---

## 3.4 Executor와 `spin()`

ROS 2 노드는 구독 메시지, 타이머, 서비스 요청 등의 작업을 **콜백 함수**로 처리한다.

Executor는 실행할 콜백이 있는지 기다리고, 준비된 콜백을 실행한다.

```text
초기화
  ↓
spin()
  ↓
메시지·타이머·서비스 요청 대기
  ↓
준비된 콜백 실행
  ↓
다시 대기
```

`rclpy.spin(node)` 또는 `rclcpp::spin(node)`을 실행하면 종료되기 전까지 콜백을 처리하는 과정이 반복된다.

따라서 “노드 하나에 루프가 계속 돌아간다”는 표현은 정확히는 다음과 같이 이해할 수 있다.

> Executor가 `spin()` 상태에서 실행할 콜백을 계속 기다리고 처리한다.

# 4. RCL과 ROS 2 프로그래밍 언어

## 4.1 RCL

**RCL(ROS Client Library)**은 ROS 2의 핵심 기능을 여러 프로그래밍 언어에서 공통적으로 사용할 수 있도록 제공하는 계층이다.

| 구분 | 설명 |
|---|---|
| `rcl` | C로 작성된 ROS 2의 기본 클라이언트 라이브러리 |
| `rclcpp` | C++에서 ROS 2 기능을 사용하기 위한 라이브러리 |
| `rclpy` | Python에서 ROS 2 기능을 사용하기 위한 라이브러리 |

ROS 2는 주로 C++과 Python을 사용하지만, Java, C#, Rust, Node.js 등 다른 언어용 클라이언트 라이브러리도 개발되어 있다.

# 5. 노드의 정의

## 5.1 노드란?

노드는 ROS 2에서 특정 기능을 수행하는 **최소 실행 단위**이다.

하나의 노드는 일반적으로 하나의 역할을 담당한다.

예시:

- 카메라 데이터를 발행하는 노드
- 센서 데이터를 처리하는 노드
- 로봇팔 목표 위치를 계산하는 노드
- 모터를 제어하는 노드
- 데이터를 화면에 표시하는 노드

각 노드는 Topic, Service, Action, Parameter 등을 이용하여 다른 노드와 데이터를 주고받는다.

---

## 5.2 패키지와 노드의 차이

| 구분 | 패키지 | 노드 |
|---|---|---|
| 의미 | ROS 2 소프트웨어를 구성하는 기본 묶음 | 실제로 실행되는 프로그램의 최소 단위 |
| 포함 내용 | 노드, 설정 파일, launch 파일, 메시지 파일 등 | 특정 기능을 수행하는 실행 프로그램 |
| 예시 | `khj_basic` | `simple_pub` |
| 관계 | 하나의 패키지에 여러 노드가 들어갈 수 있음 | 노드는 특정 패키지에 포함되어 실행됨 |

```text
khj_basic 패키지
├── simple_pub 노드 또는 실행 파일
├── 다른 Python 노드
├── 설정 파일
└── launch 파일
```

> 패키지는 프로그램을 정리하고 배포하기 위한 단위이며, 노드는 실제로 실행되는 기능 단위이다.

# 6. ROS 2 통신 방식

## 6.1 Topic

Topic은 발행자와 구독자가 메시지를 주고받는 통신 방식이다.

```text
Publisher → Topic → Subscriber
```

### 특징

- 비동기식 통신
- 지속적으로 발생하는 데이터에 적합
- `1:N`, `N:1`, `N:N` 통신 가능
- 센서 데이터, 속도 명령, 로봇 상태 등에 사용

예시:

```text
/turtle1/cmd_vel
/scan
/camera/image
```

---

## 6.2 Service

Service는 요청을 보내면 응답을 돌려받는 통신 방식이다.

```text
Service Client → Request → Service Server
Service Client ← Response ← Service Server
```

### 특징

- 요청과 응답으로 구성
- 비교적 짧은 작업에 적합
- 특정 기능을 한 번 실행하고 결과를 받을 때 사용
- 서버가 응답할 때까지 클라이언트가 기다리는 형태로 사용할 수 있음

예시:

- 거북이 생성
- 배경 초기화
- 장치 설정 변경

---

## 6.3 Action

Action은 시간이 오래 걸리는 작업을 수행하기 위한 통신 방식이다.

```text
Action Client → Goal → Action Server
Action Client ← Feedback ← Action Server
Action Client ← Result ← Action Server
```

### 구성 요소

1. **Goal:** 수행할 목표
2. **Feedback:** 작업 진행 상태
3. **Result:** 작업 완료 결과

### 특징

- 작업 도중 진행 상태를 받을 수 있다.
- 실행 중인 목표를 취소할 수 있다.
- 로봇 이동, 로봇팔 동작, 내비게이션처럼 시간이 걸리는 작업에 적합하다.

---

## 6.4 Topic, Service, Action 비교

| 구분 | Topic | Service | Action |
|---|---|---|---|
| 기본 형태 | Publisher/Subscriber | Request/Response | Goal/Feedback/Result |
| 통신 특성 | 비동기 | 요청·응답 | 장시간 작업 |
| 중간 진행 상태 | 없음 | 없음 | 있음 |
| 취소 기능 | 없음 | 일반적으로 없음 | 있음 |
| 사용 예 | 센서 데이터 | 설정 변경 | 로봇 이동 |

# 11. `setup.py`에 실행 파일 등록

Python 파일을 `ros2 run`으로 실행하려면 `setup.py`의 `console_scripts`에 등록해야 한다.

```python
entry_points={
    'console_scripts': [
        'simple_pub = khj_basic.simple_pub:main',
    ],
},
```

## 11.1 등록 형식

```text
실행 파일 이름 = Python 모듈 경로:실행 함수
```

현재 설정을 분리하면 다음과 같다.

| 부분 | 값 | 의미 |
|---|---|---|
| 실행 파일 이름 | `simple_pub` | `ros2 run`에서 사용할 이름 |
| 패키지 모듈 | `khj_basic` | 내부 Python 패키지 폴더 |
| Python 파일 | `simple_pub` | `simple_pub.py` |
| 실행 함수 | `main` | 파일 안의 `main()` 함수 |

따라서 다음 명령으로 실행할 수 있다.

```bash
ros2 run khj_basic simple_pub
```

# 12. `package.xml`

`package.xml`에는 패키지에 대한 기본 정보가 들어 있다.

현재 주요 내용:

```xml
<name>khj_basic</name>
<version>0.0.0</version>
<maintainer email="khjoo990408@gmail.com">khjoo</maintainer>

<export>
  <build_type>ament_python</build_type>
</export>
```

### 주요 태그

| 태그 | 의미 |
|---|---|
| `<name>` | 패키지 이름 |
| `<version>` | 패키지 버전 |
| `<description>` | 패키지 설명 |
| `<maintainer>` | 패키지 관리자 |
| `<license>` | 라이선스 |
| `<depend>` | 빌드와 실행에 필요한 의존성 |
| `<test_depend>` | 테스트에 필요한 의존성 |
| `<build_type>` | 패키지 빌드 형식 |

현재 `simple_pub.py`는 `rclpy`를 사용하지 않으므로 `rclpy` 의존성이 등록되어 있지 않다.

나중에 실제 ROS 2 Python 노드를 작성한다면 다음과 같은 의존성이 필요하다.

```xml
<exec_depend>rclpy</exec_depend>
```

또는 패키지 생성 단계에서 다음처럼 지정할 수 있다.

```bash
ros2 pkg create \
  --build-type ament_python \
  --dependencies rclpy \
  패키지이름
```

# 18. 현재 `simple_pub.py`는 실제 ROS 노드인가?

현재 코드:

```python
def main():
    print("첫번째 프로그램입니다.")
```

이 코드는 ROS 2 패키지 안에 들어 있고 `ros2 run`으로 실행할 수 있지만, 아직 다음 기능을 사용하지 않는다.

- `rclpy.init()`
- `Node` 클래스
- Publisher
- Subscriber
- Service
- Action
- Parameter
- `rclpy.spin()`

따라서 현재 프로그램은 엄밀히 말하면 다음과 같다.

> ROS 2 패키지에 등록된 Python 실행 파일이지만, 아직 ROS 그래프에 등록되는 실제 ROS 2 노드는 아니다.

현재 프로그램을 실행해도 다음 명령에 노드 이름이 나타나지 않는다.

```bash
ros2 node list
```

# 19. 실제 ROS 2 Python 노드의 기본 형태

실제 ROS 2 노드는 대략 다음 구조를 가진다.

```python
import rclpy
from rclpy.node import Node


class SimpleNode(Node):

    def __init__(self):
        super().__init__('simple_node')
        self.get_logger().info('Simple node started.')


def main(args=None):
    rclpy.init(args=args)

    node = SimpleNode()
    rclpy.spin(node)

    node.destroy_node()
    rclpy.shutdown()


if __name__ == '__main__':
    main()
```

이 코드에서는 다음 과정이 수행된다.

```text
ROS 2 초기화
    ↓
노드 생성
    ↓
Executor에서 콜백 대기 및 처리
    ↓
노드 종료
    ↓
ROS 2 종료
```

# 20. `.gitignore`에 제외할 항목

Python을 실행하면 `__pycache__` 폴더와 `.pyc` 파일이 자동으로 만들어질 수 있다.

이 파일은 소스 코드가 아니므로 일반적으로 GitHub에 올리지 않는다.

프로젝트 최상위 `.gitignore`에 다음 내용을 추가할 수 있다.

```gitignore
# Python cache
__pycache__/
*.pyc

# ROS 2 build results
build/
install/
log/
```

이미 Git에 등록된 `__pycache__`를 제거할 때는 프로젝트 루트에서 다음 명령을 사용할 수 있다.

```bash
find . -type d -name "__pycache__" -prune -exec rm -rf {} +
git add .
git commit -m "Remove Python cache files"
```


실행:

```bash
ros2 run khj_basic simple_pub
```

왼쪽의 `simple_pub`이 `ros2 run`에서 사용하는 실행 파일 이름이다.

# 23. 수업 핵심 정리

1. ROS 2 소프트웨어 구성의 기본 단위는 **패키지**이다.
2. 실제 기능을 수행하는 최소 실행 단위는 **노드**이다.
3. 하나의 패키지에는 여러 개의 노드와 설정 파일이 들어갈 수 있다.
4. ROS 2는 RMW 계층을 통해 DDS 통신 미들웨어를 사용한다.
5. Topic은 지속적인 데이터, Service는 요청·응답, Action은 장시간 작업에 적합하다.
6. Python 패키지는 `ament_python` 형식으로 생성한다.
7. Python 파일을 `ros2 run`으로 실행하려면 `setup.py`의 `console_scripts`에 등록해야 한다.
8. 소스 코드 작성 후 `colcon build`와 `source` 과정이 필요하다.
9. 현재 `simple_pub.py`는 ROS 2 실행 파일이지만, 아직 `rclpy`의 `Node`를 생성하지 않으므로 실제 ROS 노드는 아니다.
10. 실제 노드는 `rclpy.init()`, `Node`, `spin()`, `shutdown()` 등의 구조를 사용한다.
