# 1. ROS 2 DDS 통신 설정

서로 다른 컴퓨터 또는 WSL과 Windows 사이에서 ROS 2 DDS 통신을 하기 위해 네트워크와 방화벽 설정을 확인해야 한다.

## 1.1 WSL 네트워크 설정

WSL 네트워크를 **Mirrored Networking** 방식으로 설정한다.

```text
WSL → Windows Mirrored Network 설정
```

Mirrored Networking을 사용하면 Windows와 WSL의 네트워크 인터페이스와 포트를 보다 직접적으로 연결할 수 있다.

---

## 1.2 방화벽 확인

ROS 2 DDS 통신이 차단되지 않도록 다음 항목을 확인한다.

```text
WSL 방화벽
Windows Defender 방화벽
```

Windows Defender 방화벽에서는 다음 방향의 통신을 확인한다.

```text
Inbound  : 외부에서 들어오는 통신
Outbound : 외부로 나가는 통신
```

> 현재 수업 메모에는 확인해야 할 설정 항목만 기록되어 있으며, 세부 설정 명령어는 기록되어 있지 않다.

# 2. Header 메시지 발행 실습

`std_msgs/msg/Header` 메시지를 이용하여 현재 시간과 문자열을 발행한다.

파일 위치:

```text
khj_ws/src/khj_basic/khj_basic/header_pub.py
```

## 2.1 `Header` 메시지 구조

`Header` 메시지는 다음 필드로 구성된다.

```text
stamp
frame_id
```

| 필드 | 역할 |
|---|---|
| `stamp` | 메시지가 생성된 시간 |
| `frame_id` | 메시지와 연결된 좌표계 또는 문자열 식별자 |

메시지 구조 확인:

```bash
ros2 interface show std_msgs/msg/Header
```

---

## 2.2 `header_pub.py`

```python
import rclpy
from rclpy.node import Node
from std_msgs.msg import Header


class Header_pub(Node):

    def __init__(self):
        super().__init__('header_pub')  # 노드 이름

        # 0.1초마다 콜백 실행 = 10 Hz
        self.create_timer(0.1, self.timer_callback)

        # Header 메시지를 /time 토픽으로 발행
        self.pub = self.create_publisher(
            Header,
            'time',
            10
        )

    def timer_callback(self):
        msg = Header()

        msg.frame_id = 'time test'
        msg.stamp = self.get_clock().now().to_msg()

        self.pub.publish(msg)

        self.get_logger().info(
            f'frame_id: {msg.frame_id}, '
            f'time: {msg.stamp.sec}.{msg.stamp.nanosec}'
        )


def main(args=None):
    rclpy.init(args=args)

    node = Header_pub()

    try:
        rclpy.spin(node)

    except KeyboardInterrupt:
        node.get_logger().info('KeyboardInterrupt')

    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

---

## 2.3 Timer 설정

```python
self.create_timer(0.1, self.timer_callback)
```

`0.1초`마다 `timer_callback()` 함수가 실행된다.

```text
1초 ÷ 0.1초 = 초당 10회
```

따라서 발행 주기는 다음과 같다.

```text
10 Hz
```

---

## 2.4 현재 시간 입력

```python

```

```text

nanosec  : 나노초
```

로그 출력:


# 3. QoS

QoS는 다음의 약자이다.

```text
QoS = Quality of Service
```

ROS 2에서 Publisher와 Subscriber 사이의 데이터 전달 방법을 설정하는 기능이다.

QoS 설정을 통해 다음 항목을 정할 수 있다.

- 데이터를 몇 개 보관할지
- 데이터 전달의 신뢰성을 얼마나 중요하게 볼지
- Subscriber가 늦게 실행되었을 때 이전 데이터를 받을지
- 정해진 시간 안에 데이터가 전달되는지 확인할지
- 데이터의 유효 시간을 얼마나 유지할지
- 노드 또는 토픽이 살아 있는지 확인할지

---

# 4. QoS History

History는 전송된 메시지를 어떤 방식으로 보관할지 설정한다.

## 4.1 `KEEP_ALL`

```python
history=QoSHistoryPolicy.KEEP_ALL
```

발행한 데이터를 모두 보관하도록 설정한다.

```text
메시지 0
메시지 1
메시지 2
메시지 3
...
```

`KEEP_ALL`을 사용할 때는 일반적으로 별도의 `depth` 값을 지정하지 않는다.

---

## 4.2 `KEEP_LAST`

```python
history=QoSHistoryPolicy.KEEP_LAST
```

가장 최근의 데이터만 보관한다.

이때 `depth` 값을 이용하여 몇 개의 메시지를 보관할지 지정한다.

예시:

```python
QoSProfile(
    history=QoSHistoryPolicy.KEEP_LAST,
    depth=10
)
```

위 설정은 최근 메시지 10개를 보관한다.

---

## 4.3 History 비교

| 설정 | 의미 |
|---|---|
| `KEEP_ALL` | 모든 데이터를 보관 |
| `KEEP_LAST` | 최근 데이터만 보관 |
| `depth` | `KEEP_LAST`에서 보관할 데이터 개수 |

---

# 5. QoS Reliability

Reliability는 메시지 전달에서 속도와 신뢰성 중 무엇을 우선할지 설정한다.

## 5.1 `BEST_EFFORT`

```python
reliability=QoSReliabilityPolicy.BEST_EFFORT
```

메시지 전달의 신뢰성보다 속도를 중요하게 설정한다.

일부 메시지가 손실되더라도 다음 메시지를 계속 빠르게 전달한다.

```text
속도 중심
```

센서 데이터처럼 새로운 데이터가 계속 들어오는 경우 사용할 수 있다.

---

## 5.2 `RELIABLE`

```python
reliability=QoSReliabilityPolicy.RELIABLE
```

메시지가 정상적으로 전달되는 것을 중요하게 설정한다.

```text
데이터 수신 중심
```

현재 QoS 실습에서는 `RELIABLE`을 사용한다.

---

## 5.3 Reliability 비교

| 설정 | 특징 |
|---|---|
| `BEST_EFFORT` | 속도를 우선하며 일부 데이터가 손실될 수 있음 |
| `RELIABLE` | 메시지 전달의 신뢰성을 우선함 |

---

# 6. QoS Durability

Durability는 Subscriber가 생성되기 전에 발행된 메시지를 보관할지 설정한다.

## 6.1 `TRANSIENT_LOCAL`

```python
durability=QoSDurabilityPolicy.TRANSIENT_LOCAL
```

Subscriber가 실행되기 전에 Publisher가 발행한 데이터를 보관한다.

```text
Publisher 먼저 실행
    ↓
메시지 발행
    ↓
Subscriber 나중에 실행
    ↓
이전에 보관된 메시지 수신
```

현재 QoS 실습에서는 `TRANSIENT_LOCAL`을 사용한다.

---

## 6.2 `VOLATILE`

```python
durability=QoSDurabilityPolicy.VOLATILE
```

Subscriber가 실행되기 전에 발행된 데이터는 전달하지 않는다.

Subscriber가 연결된 이후에 발행되는 새로운 데이터만 수신한다.

---

## 6.3 Durability 비교

| 설정 | 의미 |
|---|---|
| `TRANSIENT_LOCAL` | Subscriber 생성 이전의 데이터도 보관 |
| `VOLATILE` | Subscriber 생성 이전의 데이터는 무효 |

```python
    QoSDurabilityPolicy,
    QoSHistoryPolicy,

    qos_profile_default,

class Qos_M_pub(Node):

        self.qos_profile = QoSProfile(
            history=QoSHistoryPolicy.KEEP_ALL,

        self.get_logger().info(msg.data)
```

```python
from rclpy.qos import (
    QoSDurabilityPolicy,
    QoSHistoryPolicy,
    QoSProfile,
    QoSReliabilityPolicy,
    qos_profile_default,
)

class M_sub(Node):

if __name__ == "__main__":
```

## 13.1 첫 번째 터미널: Publisher 실행

```bash
ros2 run khj_basic qos_test_pub
```

Publisher는 1초마다 `/message` 토픽으로 문자열을 발행한다.

예상 출력:

```text
[INFO] [massage_pub]: 첫번째 프로그램입니다. 0
[INFO] [massage_pub]: 첫번째 프로그램입니다. 1
[INFO] [massage_pub]: 첫번째 프로그램입니다. 2
```

---

## 13.2 두 번째 터미널: Subscriber 실행

Publisher를 먼저 실행하여 메시지를 여러 개 발행한 뒤 Subscriber를 실행한다.

```bash
ros2 run khj_basic qos_test_sub
```

예상 출력:

```text
[INFO] [massage_sub]: 첫번째 프로그램입니다. 0
[INFO] [massage_sub]: 첫번째 프로그램입니다. 1
[INFO] [massage_sub]: 첫번째 프로그램입니다. 2
```

`TRANSIENT_LOCAL`과 `KEEP_ALL`이 적용되어 있기 때문에 Subscriber를 나중에 실행하여 이전에 보관된 메시지를 확인한다.
